# V2X Robot Setup Guide

> **Last updated:** 2026-06-14
> **Hardware:** Raspberry Pi 5, Ubuntu 24.04 LTS, Arducam B0390 / IMX219 camera
> **Stack:** Pure Python — no ROS2 needed.

---

## Quick Start (the short version)

```
1. Flash Ubuntu 24.04 to SD card
2. Boot Pi, update, clone repo
3. git config --global user.name "Your Name"
   git config --global user.email "you@email.com"
4. Edit 1 line in the OBU config (RSU / Desktop IP)
5. sudo bash setup.sh car        ← does everything automatically
6. sudo reboot
7. v2x_run_car                   ← clean start (regenerates auth config + keys)
```

---

## Step 1 — Flash Ubuntu 24.04

1. Download **Raspberry Pi Imager**
2. Choose **Ubuntu Server 24.04 LTS (64-bit)**
3. In Imager advanced settings (before flashing):
   - Hostname: `v2x` (car) or `v2x-emgy` (ambulance) — must be unique per device
   - Enable SSH
   - WiFi credentials
   - Username: `veerobot`
4. Flash to microSD (32 GB+), insert, power on
5. Wait ~90 seconds, then SSH in:

```bash
ssh veerobot@v2x.local
sudo apt update && sudo apt upgrade -y
```

## Step 4 — Set the laptop / RSU IP (the one thing you edit)

Both the car and ambulance read the laptop IP from the same file:

```bash
nano ~/projects/V2X/v2x_testbed/obu/config/obu1_config.json
```

Set `rsu_ip` and `desktop_ip` to your **laptop's WiFi IP** (same value for both):

```json
{
    "rsu_ip":     "192.168.x.x",
    "desktop_ip": "192.168.x.x",
    ...
}
```

That is the only file you need to edit. Everything else (`obu_local.json`, entity IDs,
`is_emergency` flag, per-device port config) is auto-generated by the run scripts.

---

## Step 5 — Run setup.sh

```bash
cd ~/projects/V2X/robot_python
sudo bash setup.sh car          # for car robot
# OR
sudo bash setup.sh ambulance    # for ambulance robot
```

First run takes **5–10 minutes** (libcamera Python bindings compile from source).
Subsequent runs skip completed steps and finish in under a minute.

| Step | Action |
|------|--------|
| 1 | Installs system packages (picamera2, pygame, cmake, etc.) |
| 2 | Adds Raspberry Pi apt repo, installs libcamera 0.5 |
| 3 | Builds Python 3.12 libcamera bindings from source *(Pi 5 + Ubuntu 24.04 requirement)* |
| 4 | Creates Python venv (`--system-site-packages`) |
| 5 | Installs pip packages (pyserial, opencv, numpy, etc.) |
| 6 | Patches picamera2 for headless operation |
| 7 | Sets `/boot/firmware/config.txt` (camera overlay, UART enable) |
| 8 | Removes serial console from `cmdline.txt` (frees UART for STM32) |
| 9 | Adds user to `dialout` group, creates `/dev/ttyRobot` udev symlink |
| 10 | Builds OBU binary; generates `obu_local.json` from hostname |
| 11 | Installs CLI commands: `v2x_run_car`, `v2x_run_ambulance`, `v2x_robot_log` |
| 12 | Installs and enables `v2x_car` or `v2x_ambulance` systemd service |

After setup, `obu_local.json` is owned by the current user so the run scripts can update it without sudo.

## Step 7 — First clean start

After the service starts, run the appropriate command to regenerate the auth config, clear any stale keys, and get a confirmed-clean session:

```bash
v2x_run_car        # car Pi
v2x_run_ambulance  # ambulance Pi
```

Expected OBU log lines:
```
[REG] ✓ Registration complete. Keys received:  ...  Peers: 1
[UDP] Bound to 0.0.0.0:5003  (car)
[UDP] Bound to 0.0.0.0:XXXXX (ambulance — ephemeral port)
[AUTH] ║   SESSION ESTABLISHED SUCCESSFULLY   ║
```

> **Both robots start DISARMED.** Press **Start (btn 7)** on the joystick to arm and begin lane following.

## Hardware Wiring

### Each robot needs

| Component | Connection |
|-----------|-----------|
| STM32F405 | UART to RPi GPIO14 (TX) / GPIO15 (RX) |
| Camera (IMX219 / Arducam B0390) | CSI ribbon to **CAM/DISP 0** port *(not port 1)* |
| RF joystick USB dongle | Any USB port |
| Battery 5V BEC | Powers the Pi |

### STM32 UART wiring

```
STM32 TX  →  RPi GPIO15  (Pin 10, UART RX)
STM32 RX  →  RPi GPIO14  (Pin 8,  UART TX)
STM32 GND →  RPi GND     (Pin 6)
```

> Shared GND only — do NOT connect STM32 power to RPi 3.3V.
> **Unplug the STM32 USB cable during normal operation** — when USB is connected, STM32 routes UART to USB and the Pi sees its own bytes back.

---

## STM32 Firmware

Flash once per board. Both robots use identical firmware.

1. Open `V2X/STM32F405RGTx` in STM32CubeIDE
2. Build (`Ctrl+B`) and flash (Run button)
3. Board waits for Pi UART — no further config needed

## Joystick Controls

| Button/Stick | Action |
|---|---|
| Press **Start** (btn 7) | Arm / Disarm toggle |
| Hold **LB** (btn 4) | Manual joystick override (autonomous pauses) |
| Hold **LB + RB** (btn 4+5) | Turbo mode (0.8 m/s) |
| Left stick Y | Forward / reverse |
| Right stick X | Steer |
| **A button** *(ambulance only)* | Activate V2X emergency signal |
| **B button** *(ambulance only)* | Cancel V2X emergency signal |

Run `jstest /dev/input/js0` to verify button/axis numbers match your controller.
Edit `config.yaml` under `joystick:` if they differ.

## V2X Network Setup

All devices must be on the **same WiFi network**.

```
Laptop (RSU + Desktop)
  ├── RSU binary: UDP 5000 (auth), TCP 8001/8002 (registration)
  └── Desktop Flask: HTTP 5000 (dashboard), TCP 9000 (log receiver)

Car Pi
  └── OBU → RSU auth → gets session key
  └── RSU alert listener: UDP 5001 ← receives EMERGENCY_ACTIVE broadcast

Ambulance Pi
  └── OBU → RSU auth with is_emergency=true → RSU broadcasts EMERGENCY_ACTIVE → car yields
```

### RSU broadcasts to subnet

The RSU sends emergency alerts to `192.168.0.255` (subnet broadcast). Every robot on the same WiFi receives them automatically regardless of IP — no per-robot RSU config needed.

### Test the V2X chain without ambulance

From the laptop, simulate an RSU alert:

```bash
python3 -c "
import socket, json, time
s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
s.sendto(json.dumps({'type':'EMERGENCY_ACTIVE','session_id':'test'}).encode(), ('192.168.x.x', 5001))
print('Alert sent'); time.sleep(5)
s.sendto(json.dumps({'type':'EMERGENCY_CLEARED','session_id':'test'}).encode(), ('192.168.x.x', 5001))
print('Clear sent')
"
```

Car should log: `V2X emergency ACTIVE → EVADING → HOLDING → RESUMING → NORMAL`

## Per-robot differences

| Setting | Car | Ambulance |
|---------|-----|-----------|
| Hostname | `v2x` | `v2x-emgy` |
| Service | `v2x_car` | `v2x_ambulance` |
| entity_id | `V2X` (from hostname) | `V2X_EMGY` (from hostname) |
| is_emergency | false | true |
| udp_listen_port | 5003 (fixed) | 0 (OS-assigned ephemeral) |
| Control socket port | 5010 | 5011 |
| Emergency handler | Yes — yields to ambulance | No — drives through |
| Starts armed? | No — press Start | No — press Start |
| Joystick A/B | Not used | A=emergency on, B=off |

## When the laptop / RSU IP changes

Only **one file** controls all network IPs: `v2x_testbed/obu/config/obu1_config.json`.
The run scripts read from it; the robot Pi IPs are irrelevant (they bind to 0.0.0.0 and
the RSU alerts via subnet broadcast).

```bash
# On a Pi — update both fields to the new laptop IP:
nano ~/projects/V2X/v2x_testbed/obu/config/obu1_config.json
# Set "rsu_ip" and "desktop_ip" to the new IP

git commit -am "update laptop IP to 192.168.x.x"
git push

# On the other Pi:
git pull && v2x_run_car       # or v2x_run_ambulance
```

**What does NOT need to change when Pi IPs change:** nothing. The robots
bind to `0.0.0.0` and the RSU sends emergency alerts to `192.168.0.255`
(subnet broadcast). A new Pi IP assigned by the router is transparent.

**When RSU moves to its own Pi** (separate from Desktop): additionally edit
`v2x_testbed/rsu/config/rsu_config.json` and change `"desktop_ip"` from
`"127.0.0.1"` to the Desktop laptop's actual network IP. Then also split
`rsu_ip` and `desktop_ip` in `obu1_config.json` to point to different machines.

## Troubleshooting

### Camera not working

```bash
cd ~/projects/V2X/robot_python && source .venv/bin/activate
python3 -c "import picamera2; cam=picamera2.Picamera2(); print(cam.camera_properties)"
```
- Check ribbon cable is in **CAM/DISP 0** port (not port 1)
- `dmesg | grep imx219` — should show `registered`, not `probe failed`
- Re-run `sudo bash setup.sh car` if libcamera was not built

### STM32 not connecting

```bash
ls -la /dev/ttyRobot     # must exist (symlink to ttyAMA10 on RPi 5)
groups                   # must include: dialout
python3 motor_test.py    # should drive wheels for 3 s
```
If `/dev/ttyRobot` missing: re-run setup or manually add the udev rule (see `setup.sh` step 9).
If telemetry shows loopback: **unplug the STM32 USB cable**.

### OBU permission denied when updating config

```bash
sudo chown veerobot:veerobot ~/projects/V2X/v2x_testbed/obu/config/obu_local.json
v2x_run_car   # or v2x_run_ambulance
```
This happens when an older `sudo bash setup.sh` left the file owned by root. The current setup.sh chowns it automatically, but old installs may need this one-time fix.

### Robot not arming

Both robots start **disarmed** — press **Start** (btn 7) to arm.
Or use the control socket:
```bash
python3 ~/projects/V2X/robot_python/control_socket.py --port 5010 arm
```

### V2X alert not received on car

```bash
sudo journalctl -u v2x_car | grep "RSU alert listener"
# Must show: RSU alert listener started on UDP port 5001
# If missing: check config.yaml  v2x_bridge: manual_mode: false
```

V2X Security

32-Step Mutual Authentication Protocol

1
Registration (one-time)
Desktop generates EC P-256 keypairs. Issues RID, AID, DAID, SK, PK_self, PK_RSU to each OBU.
2
AuthRequest — Ambulance OBU → RSU
PID | timestamp | ECDH_encapsulate(PK_RSU) | ECDSA_sign(SK)
3
RSU validates + derives session keys
Timestamp check ±500ms → ECDSA verify → ECDH decapsulate → HKDF → SK_enc + SK_mac
4
Mutual key confirmation KC1 / KC2
Both sides prove they derived the same session keys.
5
Post-auth message (AES-256-GCM encrypted)
{"entity_id":"V2X_EMGY","is_emergency":true}
RSU decrypts → broadcasts EMERGENCY_ACTIVE to all robots.

Emergency Response

5-State Yield Sequence

NORMAL
→
EVADING
→
HOLDING
→
RECOVERING
→
RESUMING
→
NORMAL
EVADING

Turns right toward inner island yellow line.
Proportional yellow-repel steering until yellow boundary detected, or 2 s timer fires.

HOLDING

Creeps slowly along inner island wall.
Exits when: ambulance zone passes car zone (+ 1s grace) or V2X clears or 30 s hard timeout (safety net).

RECOVERING

Arcs left back toward white line.
Gyro mode: exits when STM32 IMU measures 30° of yaw. Timer fallback if IMU is noisy.

RESUMING

Ramps speed from 0 → normal over 2 s.
Smooth re-entry to full-speed lane following. No jerk.

Key insight: while in HOLDING, the car can't see inner tags (it's on the inner island). It uses its frozen last-known zone + the ambulance's live UDP zone stream to detect when the ambulance passes. When diff = (own_zone − amb_zone) % 10 > 5 → ambulance has passed.

Future-Proofing

Post-Quantum Cryptography — Interface Ready

Classical ECDSA and ECDH are broken by Shor's algorithm on a sufficiently large quantum computer. This platform's cryptographic layer is fully abstracted — switching to post-quantum is a one-line config change.

Pluggable crypto provider

// crypto_provider.h — pure virtual interface virtual encapsulate(); virtual decapsulate(); virtual sign(); virtual verify(); virtual derive_keys(); // rsu_config.json — to switch provider: "crypto_provider": "lattice" // was: "placeholder"

Planned algorithms

CRYSTALS-Dilithium — replaces ECDSA signatures
CRYSTALS-Kyber — replaces ECDH key encapsulation
Both are NIST PQC standardised (FIPS 204 / FIPS 203)

What doesn't change when you switch

The 32-step protocol — identical
Robot code — doesn't know which provider is active
Emergency broadcast format — unchanged
Dashboard — unchanged
Position sharing — unchanged (plain UDP, unrelated to crypto)

Other swappable layers

Lane algorithmcentroid / pure pursuit / recorded-path — one line in config.yaml
Motor controllerLyra protocol over UART — retarget firmware without touching Python
AprilTag layoutn_inner_tags / n_outer_tags in config — yield logic adapts
Crypto providerplaceholder (ECDH) → lattice (Kyber) — one JSON key

Geometry-based RECOVERING — integrate gyro_z from STM32 to measure actual degrees turned (target: 30°), not time elapsed.
2
Dead-reckoning between tags — continuous position from wheel encoder ticks; resets on every AprilTag sighting.
3
Variable crop ratio by state — NORMAL: 40%, EVADING: 20%, RECOVERING: 0%. Wider view during critical manoeuvres.
4
Post-quantum — CRYSTALS-Kyber + Dilithium — plug into existing LatticeProvider interface.

No new hardware required for any of the first three improvements. The STM32 already sends gyro_z and wheel_ticks[4]. The hard part is done.
